## Introduction

The Servineo API is built using a modern RESTful architecture with Redux Toolkit Query (RTK Query) for client-side data fetching and caching. The API powers the Servineo platform, connecting service providers (fixers) with customers (requesters).

## Architecture

### Base Configuration

The API uses a centralized configuration approach with RTK Query:

```typescript ~/workspace/source/src/app/redux/services/baseApi.ts
import { createApi, fetchBaseQuery } from '@reduxjs/toolkit/query/react';

const API_URL = process.env.NEXT_PUBLIC_API_URL + '/api';

export const baseQuery = fetchBaseQuery({
  baseUrl: API_URL,
  credentials: 'include',
});

export const baseApi = createApi({
  reducerPath: 'api',
  baseQuery,
  tagTypes: [
    'User',
    'Job',
    'Statistics',
    'JobOffer',
    'Requester',
    'SearchHistory',
    'Experience',
    'Portfolio',
    'Certification',
  ],
  endpoints: () => ({}),
});
```

### Base URL

All API endpoints are relative to:

```
{NEXT_PUBLIC_API_URL}/api
```

**Example:**
- Development: `http://localhost:8000/api`
- Production: Configured via environment variables

## Core Concepts

### RTK Query

Servineo uses Redux Toolkit Query for API interactions, providing:

- **Automatic caching**: Reduces unnecessary network requests
- **Automatic refetching**: Keeps data fresh with tag-based invalidation
- **Loading states**: Built-in loading, error, and success states
- **Optimistic updates**: Immediate UI updates before server confirmation

### Tag-Based Cache Invalidation

## Error Handling

### Error Response Format

All API errors follow a consistent structure:

```typescript
export interface ApiError {
  status: number;
  data: {
    message: string;
    errors?: Record<string, string[]>;
  };
}
```

### Type Guard

Use the provided type guard to check for API errors:

```typescript
import { isApiError } from './baseApi';

try {
  const result = await createJob(formData);
} catch (error) {
  if (isApiError(error)) {
    console.error(error.data.message);
    // Handle validation errors
    if (error.data.errors) {
      Object.entries(error.data.errors).forEach(([field, messages]) => {
        console.error(`${field}: ${messages.join(', ')}`);
      });
    }
  }
}
```

## Request Headers

### Authentication

<Note>
Authentication is handled via HTTP-only cookies with `credentials: 'include'` configuration.
</Note>

For endpoints requiring Bearer tokens:

```typescript
headers: {
  'Authorization': `Bearer ${token}`,
  'Content-Type': 'application/json'
}
```

## Response Formats

### Standard Success Response

```typescript
interface ApiResponse<T> {
  success: boolean;
  data?: T;
  message?: string;
}
```

### Paginated Response

Some endpoints return paginated data:

```typescript
interface PaginatedResponse<T> {
  data: T[];
  count: number;
  success?: boolean;
}
```

## Rate Limiting

The API implements rate limiting for security:

- **2FA verification**: Limited attempts before account lock (423 status)
- **Lock duration**: Returned in `retryAfterSeconds` field
- **Remaining attempts**: Returned in `attemptsLeft` field
